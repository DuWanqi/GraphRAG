# 回忆录 RAG 系统汇报流程图

> 日期：2026-04-16

## 📋 开场：一句话概要

```
用户输入跨越数十年的回忆录
    ↓
按时间边界分段
    ↓
逐段检索知识图谱 → 注入跨章上下文 → LLM生成
    ↓
多维度评估 + 质量门控
    ↓
输出完整润色长文（不达标自动重试）
```

**核心价值**：保留原文细节 + 融入历史背景 + 自动质量控制

---

## 🎯 第一部分：需求与挑战

### 需求
```
输入：3500字回忆录，跨越1972-2008年
      包含个人经历（知青、高考、创业、金融危机）

期望：生成润色长文
      ✓ 保留原文人物/对话/细节
      ✓ 融入历史背景（知识图谱）
      ✓ 各章连贯不重复
      ✓ 可交付质量判定
```

### 三大挑战
```
❶ 如何分段？
   → 按空行切？会把1972年和1977年合并
   → 按长度切？会破坏时间连续性
   
❷ 如何生成？
   → 各章独立生成？会重复、套话、风格割裂
   → 如何融入历史背景？直接罗列实体名？
   
❸ 如何判定质量？
   → 只有指标分数？无法回答"能不能交付"
   → 不达标怎么办？人工修改？
```

```
│  ┌─────────────────────────────────────────────────┐    │

                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│  STEP 3：逐章生成 (long_form_orchestrator.py)            │

│  └──────────────────────────────────────────────────┘   │
│                     ↓                                     │

│  │     ③ 位置指令（开篇/中间/结尾差异化）            │   │

│                     ↓                                     │
│  ┌──────────────────────────────────────────────────┐   │

│                     ↓                                     │
│  ┌──────────────────────────────────────────────────┐   │

│  └──────────────────────────────────────────────────┘   │
│                     ↓                                     │
│  ┌──────────────────────────────────────────────────┐   │
│  │ 3e. 记录本章                                      │   │
│  │     提取：摘要（首句）+ 年份范围 + 高频短语       │   │
│  │     供下一章使用                                  │   │

│                                                           │

                     ▼

│  │  entity_coverage, time_consistency,               │   │
│  │  keyword_overlap, semantic_similarity,            │   │

│  │  transition_usage, descriptive_richness           │   │
│  │  → 综合分 = 加权平均 × 10                        │   │
│  └──────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────┐   │

│  │  → FActScore = 被支持事实数 / 总事实数           │   │
│  └──────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────┐   │

│  ┌──────────────────────────────────────────────────┐   │
│  │ 实体分析（新增）                                  │   │

│  └──────────────────────────────────────────────────┘   │
│                                                           │
│  并行化：各章评估用 asyncio.gather 并发执行               │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│  STEP 5：质量门控 (quality_gate.py)                      │
│                                                           │
│  逐章检查 7 个维度：                                      │

│  │ ① 字数：实际/目标 < 0.4 或 > 2.5 → error         │   │
│  │ ② 综合分：< 5.0 → error                          │   │

│  │ ④ 总结句：占比 > 30% → warning                   │   │

│  │ ⑥ 感悟收尾（非末章）：命中 → warning             │   │
│  │ ⑦ 跨章重复：> 20% → error                        │   │

│                                                           │
│  判定：任一 error → passed = False                        │
│        → 生成 RemediationPlan（哪几章重做/原因/建议）     │
│                                                           │
│  自动重试闭环（最多2轮）：                                │
│    重新检索 + 构建上下文 + 追加修复指令                   │
│    → 重新生成（temperature +0.05）                        │
│    → 替换旧结果 → 重新评估                                │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐

│  ✓ 总结句 0%                                             │
│  ✓ 实体分析：RAG覆盖率 20-40%，原文细节保留 70-80%       │
└─────────────────────────────────────────────────────────┘
```

## 💡 第三部分：关键技术点

### 技术点 1：时间边界保护

```
问题：按空行切分会把不同年代合并
      1972年200字 + 1977年300字 → 合并成一段？

方案：不跨时间边界合并
      if 当前段太短 and not 下一段是新年份:
          合并
      else:
          不合并（即使只有200字）

效果：保证每段时间精度 → 检索更准确
```

**讲解要点**：
- 展示代码片段：`crosses_time = _LEADING_YEAR.match(b)`
- 强调：时间精度 > 段落长度均衡

---

### 技术点 2：跨章上下文管理

```
问题：各章独立生成 → 重复、套话、风格割裂

方案：ChapterContext 注入三块内容
      ① 前文概要（最近3章摘要）
      ② 反重复要点（jieba分词+停用词过滤）
      ③ 位置指令（开篇/中间/结尾差异化）

效果：各章连贯，跨章重复率 0.2%
```

**讲解要点**：
- 展示 Prompt 结构：前文概要 + 反重复要点
- 对比：字符滑窗（有效率37.5%）vs jieba分词（有效率100%）

---

### 技术点 3：实体分析（RAG vs 文本）

```
发现：entity_coverage 普遍很低（0-20%）
      但综合评分却有 8.92

原因：RAG实体（宏观历史）vs 文本实体（个人叙事）
      在不同语义层面

      RAG检索：恢复高考、邓小平南巡、亚洲金融危机
      生成文本：老张、德明、林老板、窑洞

方案：分别列出两类实体 + 覆盖关系
      rag_retrieved: [恢复高考, 邓小平, ...]
      rag_used_in_text: [恢复高考]
      text_extracted: [老张, 德明, ...]
      text_only_entities: [老张, 德明, ...]

结论：RAG覆盖率低（20%）≠ RAG无效
      而是以背景方式融入，而非罗列实体名
```

**讲解要点**：
- 展示第1章和第4章的实体对比
- 强调：好的生成 = 保留原文细节（70-80%）+ 融入关键历史事件（1-2个）

---

### 技术点 4：FActScore 双证据源

```
问题：只用检索背景验证 → 原文人名/对话被判"无据"
      FActScore 持续偏低（~45%）

方案：验证上下文 = 回忆录原文 + 检索背景
      + 文学修辞过滤（"仿佛黄土高坡在低吟"不验证）
      + 规则预匹配（≥50%关键词在原文 → 直接支持）

效果：FActScore 从 ~45% 提升到 ~70%
```

**讲解要点**：
- 展示三步流程：分解 → 过滤 → 验证
- 强调：双证据源的必要性

---

### 技术点 5：质量门控 + 自动重试

```
问题：只有评分，无法回答"能不能交付"

方案：7维度检查 → passed: true/false
      + RemediationPlan（哪几章重做/原因/建议）
      + 自动重试闭环（最多2轮）

效果：第1轮重试第5章，第2轮重试第4章
      达到最大重试次数后给出明确报告
```

**讲解要点**：
- 展示质量门控的7个维度
- 展示重试日志：3轮重试记录

## 🎓 第五部分：核心改进总结

### 改进前的三个问题

| 问题 | 具体现象 |
|---|---|
| ① 分段不按年代切 | 1972年和1977年被合并，检索精度下降 |
| ② 各章独立生成 | 重复、套话、风格割裂 |
| ③ 无可交付判定 | 只有指标分数，无法回答"能不能交付" |

### 改进后的方案

| 问题 | 方案 | 效果 |
|---|---|---|
| ① 分段 | 时间边界保护 + 不跨年代合并 | 100%按时间切分 |
| ② 生成 | ChapterContext + jieba分词反重复 + 位置指令 | 跨章重复率0.2% |
| ③ 质量 | 质量门控 + 修复计划 + 自动重试 | 明确可交付判定 |

### 新增：实体分析

| 改进前 | 改进后 |
|---|---|
| 只有 entity_coverage（0-20%） | 分别列出 RAG实体 vs 文本实体 |
| 无法区分实体类型 | 清晰展示覆盖关系 |
| 容易误判"RAG没用上" | 准确判断RAG作用方式 |
